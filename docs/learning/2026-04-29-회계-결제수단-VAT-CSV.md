## 📚 새로 배운 / 정리한 개념

### 1) 한국 소형 매장 회계 — 부가세 정책

| 항목 | 내용 |
|---|---|
| 일반과세자 | 부가세 10%. 매출 4,800만원 이상. 분기별 신고. |
| 간이과세자 | 부가세 ~4%. 연 매출 4,800만 미만. 연 1회 신고. |
| 가격 표시 | **소비자 가격 = 부가세 포함** 이 한국 표준. 메뉴판 가격 = 받을 돈. |
| 신고 시 분리 | 공급가액 = 매출 ÷ 1.1, 부가세 = 매출 - 공급가액. |

**채택**: 부가세 10% 가정 (일반과세자 우선). 부가세 포함 가격 입력. RevenueScreen 의 토글로 공급가액 / 부가세 분리 표시. 매장이 간이과세면 표시값 참고만 하고 신고 시 자체 처리.

### 2) 부가세 분리 — 정수 반올림 정책 단일화

```js
// total = 1000 인 경우
// 1000 / 1.1 = 909.09... → 909 (반올림)
// 1000 - 909 = 91
// 합 = 1000 ✓
splitVatIncluded(1000) // { total: 1000, supply: 909, vat: 91 }
```

**핵심**: `Math.round(total / 1.1)` 로 공급가액 먼저, `vat = total - supply` 로 차감. 이렇게 하면 **합 = total 항상 보장**. 두 값을 각각 round 하면 합이 1원 어긋날 수 있음 (사장님 신고 합계가 1원 안 맞으면 의심당함).

Jest 케이스로 강제: `expect(supply + vat).toBe(total)`.

### 3) CSV — Excel 한글 BOM

UTF-8 CSV 를 Excel 에서 그냥 열면 한글이 깨짐. 파일 선두에 BOM(`﻿`) 추가하면 Excel 이 UTF-8 로 인식.

```js
const bom = '﻿';
const blob = new Blob([bom + text], { type: 'text/csv;charset=utf-8' });
```

### 4) CSV escape — 표준 RFC 4180

셀에 `,` `"` `\n` 있으면 큰따옴표로 감싸고 내부 `"` 는 `""` 두 번.
- `치킨×2,콜라×1` → `"치킨×2,콜라×1"`
- `메뉴 "특별"` → `"메뉴 ""특별"""`

### 5) `summarizeDaily` — 한 번 순회로 메뉴/시간대/결제수단 모두 집계

매장 1일 history 가 수백건 단위. 3번 순회보다 1번 순회가 효율적이고 코드 일관됨. byMenu 는 Map → array 정렬, byHour 는 24개 fixed array, byPayment 는 별 함수 위임.

### 6) `.web.js` / `.js` 분리 패턴 누적 사례

오늘만 3건 추가 (csvDownload, otaUpdates, pwaSetup). 누적 5건 (+ sentry, firebase). 패턴이 안정적 — 네이티브/웹 분기 필요한 모듈은 일단 이렇게.

### 7) 결제수단 4종이 한국 소형 매장 현실 커버

- **현금** — 카드 안 받는 곳도 있음
- **카드** — 가장 흔함. PG 수수료 정산 시 별도 빼야
- **계좌이체** — 단골 / 대량 주문
- **지역화폐** — 정부 지원 매출. 별도 정산 가능

5번째? 카카오페이/네이버페이/제로페이 등 모바일 — 매장 환경에서는 결국 "카드" 또는 "계좌이체" 로 들어옴. UI 4종으로 충분.

## 🐛 발견한 함정 + 해결

### 함정 — markPaid 와 clearTable 의 paymentMethod 우선순위

- 선불 시: markPaid 가 paymentMethod 슬롯에 저장
- 후불 시: clearTable 가 직접 paymentMethod 받아 history 에 박음
- 후불 단계에서 안 받으면 → slot 의 옛 값 (선불 단계에서 저장됐을 수도) 사용
- 둘 다 없으면 null = 미분류

```js
paymentMethod: paymentMethod || existing.paymentMethod || null
```

이 fallback 체인으로 자연스럽게 처리. 사용자가 "선불 후 후불로 다시 정리" 같은 비정상 흐름 가도 옛 값 유지.

### 함정 — Metro 모듈 그래프 캐시 (워크트리 환경)

이전 세션 OTA 셋업 중 발견된 metro-runtime vendor 누락 이슈가 본 세션에서도 영향. vendor 폴더 복사해도 Metro 가 그래프 캐시 보유 — 서버 재시작 필요.

**결정**: 워크트리 dev preview 검증은 포기하고 Jest + 본관 검증으로 대체. 우리 회계 코드는 181 테스트 통과로 견고.

## 🧠 자기 점검 — 다음에 떠올릴 것

1. **금액 / 부가세 / 합계 같은 회계 숫자는 단일 라운딩 정책 + 합 = total 검증** — Jest 케이스로 강제. 1원 어긋남이 신고 의심으로 이어짐.

2. **CSV 는 BOM + escape 두 가지가 표준** — Excel 한글 안 깨짐 + 셀에 콤마/따옴표 들어가도 안전. 한 번 만들어두면 회계 사무소 송부 영구 사용.

3. **paymentMethod 같은 schema 추가는 fallback 체인 필수** — 옛 데이터 (`null`) → 미분류, 새 데이터 → 선택값. 마이그레이션 없이 호환.

4. **결제 진입점 정리 — 한 곳에 모이게** — markPaid + clearTable 두 곳뿐. picker UI 만 추가하면 모든 결제 흐름 커버. 진입점이 흩어져 있으면 새 필드 추가 시 누락.

5. **순수 함수 + Jest 단위 테스트가 회계처럼 정확성 핵심인 코드의 기본 도구** — UI 검증 안 돼도 함수는 견고. 패턴이 누적된 효과 (orderHelpers / storeDiag / payment 세 사례).

6. **utils/payment.js 의 함수들이 향후 재사용 자산** — 월계 보고서 / 연간 합계 / 카테고리별 분석 등 다음 분석 화면도 여기 함수 위에서 1-2시간이면 추가 가능.
