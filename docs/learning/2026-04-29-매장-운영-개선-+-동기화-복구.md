## 📚 새로 배운 개념

### 1) Firebase 익명 uid 는 (기기 + 브라우저) 단위

**같은 사용자라도** 각 기기 / 브라우저 / 데이터 클리어 단위로 uid 다름. 매장 멤버십도 uid 별이라:

| 상황 | 가입 절차 필요? |
|---|---|
| 새 직원 폰 | ✓ |
| 카운터 PC 교체 | ✓ |
| 같은 PC 의 다른 브라우저 | ✓ |
| 같은 PC 의 시크릿 창 | ✓ (시크릿 닫으면 또 잃음) |
| 같은 PC, 같은 브라우저, 캐시 클리어 | ✓ (IndexedDB 의 익명 uid 도 날아감) |
| 같은 PC, 같은 브라우저, 계속 사용 | ✗ 한 번 가입하면 영구 |

매장 운영 매뉴얼에 추가 필요.

### 2) Firestore 보안 규칙의 owner 검증 vs 임시 우회

```js
function isOwner(storeId) {
  return isMember(storeId)
    && get(/databases/$(database)/documents/stores/$(storeId)/members/$(uid())).data.role == 'owner';
}

match /joinRequests/{requestUid} {
  allow read: if requestUid == uid() || isOwner(storeId);  // owner 만 모든 요청 read
}
```

만약 사장님 폰의 익명 uid 가 owner role 로 등록 안 되어있으면 → joinRequests listener 가 0건 read → 가입 요청 화면에 안 뜸.

**해결 우선순위**:
1. 매장 운영 우선 → 임시 `if true` 로 풀기
2. 운영 안정 후 owner 권한 진단 + 운영 규칙 복원

### 3) Firestore 기반 매장 공유 이벤트 패턴

음성 동기화 같은 "한 기기 액션 → 모든 기기 반응" 패턴:

```
[기기 A]   action 발생
   ↓
   1) 본인 즉시 처리 (네트워크 왕복 대기 X)
   2) Firestore 의 단일 문서에 이벤트 박기
       (stores/{sid}/state/audioEvent)
   ↓
[Firestore]  변경 감지
   ↓
[기기 B,C,D] onSnapshot listener 가 받음
   ↓
   source === self ? skip (본인 이미 처리)
   : 본인도 같은 처리 실행
```

**핵심**:
- `source` 필드로 본인 이벤트 구별 → 중복 방지
- `ts` 필드로 옛 이벤트 무시 (mount 시점 cutoff)
- 본인 즉시 처리 → 네트워크 왕복 50ms 대기 X

이 패턴은 **음성 / 알림 / 강제 새로고침 / 일제 액션** 등 광범위 응용 가능.

### 4) PC 카운터 키오스크 운영 — UX 작은 차이의 큰 영향

| 기존 | 개선 |
|---|---|
| 영구 깜빡임 (사용자 보는 동안 계속) | 3초 후 자동 소멸 |
| 결제완료 = 테이블 진입 → 선불/후불 (2단계) | "결제하기" 한 번 = markPaid + clearTable (1단계) |
| 음성: 트리거 기기만 들음 | 모든 매장 멤버 동시 들음 |

매장 운영 흐름 = **터치 횟수 / 시각적 피로 / 청각 분리** 가 본질. UX 작은 차이가 영업 효율 큰 차이.

## 🐛 발견한 함정 + 해결

### 함정 1 — 새 PC 에서 매장 가입 화면 안 뜸 (옛 빌드 캐시)

**증상**: 한 번도 가입한 적 없는 새 PC 에서 라이브 URL 접속 → 매장 가입 화면 X 메인 화면 직접.

**진단**:
- USE_GATE 가 false = `process.env.EXPO_PUBLIC_FIREBASE_API_KEY` 가 undefined
- 빌드 산출물 grep 결과 inline 안 됨
- → 옛 빌드 (env inline 안 된 시기) 가 라이브에 올라가있던 것

**해결**: 본관에서 클린 빌드 + 재배포 (`rm -rf dist + npx expo export --platform web + npx wrangler deploy`).

**근본 원인 추정**: 본관 폴더에서 누군가/언제인가 빌드를 한 번 더 돌려서 옛 빌드를 만든 후 그게 라이브에 올라감. 정확한 시점 모름.

**방지책**: `package.json` 에 `npm run deploy:web` 같은 정식 스크립트 + 빌드 후 항상 `grep "AIzaSy" dist/.../*.js` 검증 한 줄.

### 함정 2 — 운영 규칙 deploy 후 가입 요청 listener 차단

**증상**: 새 PC 가입 요청 → 사장님 폰에 안 보임.

**원인 (추정)**: 옛 매장의 사장님 폰 익명 uid 가 owner 가 아니거나, role 필드 값이 'owner' 가 아닌 다른 값. 또는 listener 자체가 어떤 이유로 비활성.

**우회**: 임시 `if true` 풀기 → 그래도 안 됨 → **새 매장 만들기**.

**근본 해결 (다음 세션)**: 옛 매장의 멤버 데이터 검증. Firebase 콘솔에서 `stores/{oldId}/members` 직접 확인.

### 함정 3 — `Alert.alert` silent on RN Web (이전 세션과 동일)

이번 세션에서도 적용된 utils/alertCompat 가 잘 동작.

## 🧠 자기 점검 — 다음에 떠올릴 것

1. **본관에서 빌드/배포 명령 표준화** — `npm run deploy:web` 같은 정식 스크립트 정의. 그 안에서 grep 검증까지.

2. **Firestore 보안 규칙 deploy 전 owner 권한 검증** — 운영 규칙 적용 시 누가 owner 인지 Firebase 콘솔 직접 확인. 잘못된 role 이면 데이터 수정 후 deploy.

3. **매장 옮김 / 새 PC 도입 시 가입 절차 1회 필요** — 운영 매뉴얼에 추가. 익명 uid 의 본질.

4. **Firestore 기반 이벤트 패턴은 강력** — 음성/알림 외 강제 새로고침, 일제 종료, 매장 공지 등 응용 가능.

5. **UX 작은 변경의 큰 영향** — 결제 1단계 단축, 깜빡임 자동 종료 등은 영업 누적 효율 큼. 사용자 피드백 반영 빠르게.
