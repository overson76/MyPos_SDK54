## 핵심 개념: 빌드 ≠ 배포

데스크톱 개발 비유:

| 단계 | RN/Expo Web | Visual Studio + Tekla 플러그인 |
|---|---|---|
| 빌드 | `expo export` → `dist/` 생성 | F6 → `bin/Release/MyPlugin.dll` 생성 |
| 배포 | `wrangler deploy` → Cloudflare 에 업로드 | Tekla `~/PluginDLL/` 에 .dll **복사** |
| 사용자 시점 | 매장 PC 가 라이브 URL 에서 새 번들 다운로드 | Tekla 재시작 시 .dll 재로드 |

**dist/ 폴더에 새 빌드 있다 ≠ 라이브 URL 도 최신.** Tekla 의 `bin/Release/MyPlugin.dll` 만 새로 빌드되어 있다고 Tekla 가 자동으로 로드하지 않는 것과 똑같음 — `~/PluginDLL/` 로 복사해야 그제서야 Tekla 가 새 .dll 을 봄.

이번 사고는 **빌드는 새 거인데 배포는 옛 거** 인 상태로 며칠 방치된 것.

## 함정 2 — `EXPO_PUBLIC_*` inline transformer

`babel-preset-expo` 의 `EXPO_PUBLIC_*` inline transformer 는 **`process.env.EXPO_PUBLIC_X` 직접 참조 패턴만 인식**. 변수에 담아서 우회하면 production 빌드에서 inline 실패 → 라이브 URL 에서 Firebase init 깨짐.

```js
// ❌ 우회 패턴 — production 빌드에서 undefined
const env = process.env;
const apiKey = env.EXPO_PUBLIC_FIREBASE_API_KEY;

// ✅ 직접 참조 — inline 됨
const apiKey = process.env.EXPO_PUBLIC_FIREBASE_API_KEY;
```

`utils/firebase.web.js` 의 `readConfig()` 가 직접 참조 패턴 사용 + 주석으로 함정 명시함. 절대 우회 패턴으로 리팩터링 X.

`scripts/deploy-web.sh` 의 grep 검증 단계가 빌드 산출물에서 키 inline 여부를 자동 확인 — 0개면 abort. 1초 만에 함정 2 잡음.

## 폰과 PC 의 환경변수 적용 시점 차이

| 기기 | 빌드 종류 | 환경변수 박히는 시점 | 라이브 URL 영향 |
|---|---|---|---|
| 아이폰 (사장님) | EAS production native | EAS 빌드 시 한 번 | 영향 X |
| 안드로이드 (직원) | EAS production native | EAS 빌드 시 한 번 | 영향 X |
| 아이패드 | EAS production native | EAS 빌드 시 한 번 | 영향 X |
| 카운터 PC (.exe) | Electron + 라이브 URL 로드 | **매번 라이브 URL 의 JS 받을 때** | **직접 영향** |
| 주방 PC (브라우저) | 라이브 URL 직접 로드 | **매번 라이브 URL 의 JS 받을 때** | **직접 영향** |

→ **라이브 URL 빌드가 깨지면 PC 만 동시에 끊긴다**. 폰은 멀쩡한 채 PC 만 이상하면 라이브 URL 의심이 1순위.
