## 📚 새로 배운 개념

### 1) PowerShell System.Drawing 으로 PNG 동적 생성

별도 npm 의존성(sharp / canvas) 없이 윈도우 내장 .NET 으로 PNG 만들 수 있음. 한글 인코딩만 주의.

| 항목 | 패턴 |
|---|---|
| 둥근 사각형 | `GraphicsPath.AddArc` 4번 + `CloseFigure()` |
| 그라디언트 | `LinearGradientBrush(rect, c1, c2, angle)` — `New-Object` 시 `-ArgumentList` 필수 |
| 한글 텍스트 | PowerShell 5.1 인코딩 함정 → `[char]0xB9C8` 같은 Unicode escape 권장 |
| 다운스케일 | 큰 사이즈(256) 렌더 → `InterpolationMode::HighQualityBicubic` 으로 작게 (favicon 48) |
| 같은 디자인 다양 사이즈 | `Draw-B2($g, $sx, $sy, $sw, $sh, $bgColor)` 파라미터화 — 1024/512/192/180 동일 함수로 |

PowerShell 5.1 함정:
- 기본 파일 인코딩 UTF-16 → Write 도구로 UTF-8 저장 시 PS 가 ANSI 로 오해. 해결: 한글 inline 대신 `[char]0xB9C8` 같은 코드포인트.
- `New-Object Type(arg, arg)` 는 C# 문법. PS 는 `New-Object Type -ArgumentList arg, arg`.
- `2>&1` 으로 native exe stderr 리다이렉트 시 `$?` 가 false 됨 — 사용 금지.

### 2) Expo SDK 50+ 의 `public/` 폴더

빌드 시 `dist/` 루트로 그대로 복사됨. 매니페스트 / 서비스 워커 / 정적 PWA 자산을 둘 곳. dev 서버도 같은 경로로 서빙.

### 3) PWA 매니페스트 동적 주입 (HTML 안 건드리고)

Expo Web 의 자동 생성 `index.html` 은 매니페스트/theme 메타 태그가 없음. 두 가지 선택지:

| 방법 | 장점 | 단점 |
|---|---|---|
| `public/index.html` 템플릿 | 한 번 셋업 후 안정 | Expo 업데이트 시 깨질 수 있음 |
| **App.js useEffect 로 동적 주입** | Expo 업데이트 무관, 모든 코드 JS | 첫 페이지 로드 약간 늦게 매니페스트 인식 |

**채택**: 동적 주입. 브라우저는 매니페스트 늦게 와도 install prompt 띄움 (검증됨).

핵심: `utils/pwaSetup.web.js` (웹 전용) + `utils/pwaSetup.js` (네이티브 no-op) 분리 — Sentry 와 같은 패턴.

### 4) Service Worker 안전 패턴 — network-first + skipWaiting

매장 운영 환경에서 SW 의 가장 큰 위험: **새 빌드 배포 후 옛 코드 잡고 있어 매장 깨짐**.

방어 패턴:
```js
// install: 옛 SW 안 기다림
self.addEventListener('install', () => self.skipWaiting());

// activate: 옛 캐시 전부 삭제 + 모든 탭 즉시 새 SW 제어
self.addEventListener('activate', (e) => e.waitUntil(async () => {
  await Promise.all((await caches.keys()).filter(k => k !== CURRENT).map(k => caches.delete(k)));
  await self.clients.claim();
}));

// fetch: network-first — 매번 네트워크 우선, 응답 받으면 캐시 갱신
event.respondWith(fetch(req).then(saveCacheClone).catch(() => cache.match(req)));
```

이 조합 = 평소엔 항상 최신 코드 + 인터넷 끊겼을 때만 캐시 폴백 + 새 빌드 배포 시 새로고침 1회로 즉시 갱신.

추가 안전 장치: dev 에서는 등록 안 함 (`__DEV__` 가드) — Metro HMR 과 SW 캐시 충돌 방지.

### 5) 익명 uid 진단의 핵심 — ownerId vs role 'owner' 멤버 uid

어제 사고 시나리오 재현:
1. 새 매장 만듦 → `stores/{sid}.ownerId = uidA`, `members/{uidA}` 생성
2. 어떤 사정으로 사장님 폰의 익명 uid 가 `uidB` 로 새로 발급됨 (브라우저 데이터 클리어 등)
3. 가입 요청 시 보안 규칙 `isOwner(storeId)` 가 `members/{uidB}` 확인 → 없음 → 차단
4. 사장님 폰 화면에 가입 요청 0건 → "왜 안 보이지?"

진단 핵심:
- **stores.ownerId** (옛 가입 시점 uid)
- **members 중 role='owner' 인 사람의 uid** (실제 권한 가진 사람)
- **현재 기기 익명 uid** (`getCurrentUid()`)

이 3개 비교로 8개 시나리오 분류 (`__tests__/storeDiag.test.js`):
- 일치 → ok
- ownerId vs owner role 불일치 → warn (옛 데이터 잔재)
- owner 0/2명+ → warn/error
- 본인 uid 가 멤버에 없음 → error (재가입 필요)

```bash
done
```

배포 abort 가 사고 1차 방어선.

## 🐛 발견한 함정 + 해결

### 함정 — PowerShell 5.1 의 한글 inline 인코딩 사고

**증상**: `gen.ps1` 실행 시 `Missing ')' in method call.` 같은 의미 불명 파싱 에러. 한글 문자가 깨진 위치를 PS 가 메서드 호출 인자로 잘못 인식.

**원인**: Write 도구 → 파일 UTF-8 저장. PS 5.1 → 파일 ANSI 로 오해. 한글 바이트 시퀀스 → ASCII 가 아닌 문자 → 토큰 파서 혼란.

**해결**: Korean inline 제거하고 `[char]0xB9C8 + [char]0xC774` (`마이`) 식 코드포인트 escape.

**근본 대안**: 파일을 UTF-8 BOM 으로 저장하면 PS 가 인식. 하지만 escape 가 더 안전 (BOM 빠진 사본이 돌아도 OK).

## 🧠 자기 점검 — 다음에 떠올릴 것

1. **순수 함수 + Jest 단위 테스트 패턴이 운영 차단성 코드의 기본** — 진단 로직처럼 라이브 환경 없으면 시각 검증 못 하는 것은 함수 분리 + 테스트가 가장 빠르고 견고. (orderHelpers / storeDiag 두 사례 누적.)

2. **dev 서버 검증 ≠ production 검증** (학습 누적) — dev 에서 SW 안 띄우는 게 자체 정책이라 production 검증은 라이브 배포 후만. 그래서 `deploy:web` 스크립트의 grep + 자산 존재 검증이 중요.

3. **아이콘 갱신은 빌드 사본 문제** — 파일 바꾸기는 1초, 폰에 반영은 EAS 빌드 + TestFlight + 사용자 업데이트. 사용자 기대 관리 필요. 웹은 deploy:web 만으로 즉시.

4. **PWA 의 핵심 가치는 "설치 가능" 보다 "오프라인 살아있음"** — PC 카운터 환경에서 인터넷 잠시 끊겨도 마지막 화면 + 진행 중 주문 유지가 매장 운영 안정성에 직결.

5. **`.web.js` / `.js` 분리 패턴 누적 사례**: sentry, pwaSetup, firebase. 네이티브/웹 동작 다르고 의존성도 분리 필요한 모듈은 이 패턴이 robust.

6. **PowerShell 한글 함정** — 한 번 당했으면 다음엔 자동으로 코드포인트 escape. 디자인 자산 생성 같은 1회성 스크립트는 escape 가 차라리 빠름.
