# 2026-06-09 (4부) — 주소록 편집/삭제가 재시작 시 부활하던 다기기 레이스 원천 제거

## 한 줄 요약

사장님 신고 "주소록 변경·저장 후 프로그램 껐다 켜면 옛 데이터가 다시 살아난다(대표: 사하자원). 어느 기기서 하든 서로 물려서". 근본 원인 = 부팅 시 **AsyncStorage 의 stale entries**가 `useOrderFirestoreSync` 의 `lastSyncedAddressEntriesRef` 와 어긋난 **false diff** 를 만들어, write effect 가 그 옛 항목을 Firestore 에 `batch.set` 으로 재push → 삭제된 entry 부활. 처방 = **entries 는 AsyncStorage 에서 복원하지 않고 Firestore(+persistentLocalCache)를 단일 진실원**으로.

## 근본 원인 — "false diff" 부활 경로 (다기기 한정)

주소록 entry 는 **Firestore(매장 공용) + 각 기기 AsyncStorage(로컬 캐시)** 이중 저장.

1. 기기 A 에서 「사하자원」 삭제 → A 의 write effect 가 `batch.delete` → Firestore 에서 정상 삭제 ✅
2. 기기 B(폰/맥북)가 그때 꺼져있거나 백그라운드 → **B 의 AsyncStorage 엔 사하자원 잔존**, listener 못 받음
3. **B 재시작** → `useOrderPersistence` 가 부팅 시 AsyncStorage 의 옛 entries(사하자원 포함)를 `setAddressBook` 으로 메모리에 올림
4. 그런데 `lastSyncedAddressEntriesRef` 는 **Firestore listener 만 갱신** → AsyncStorage hydrate 가 만든 entries 는 ref(진실)와 어긋난 **false diff**
5. `useOrderFirestoreSync` 의 addressBook.entries write effect: `next(로컬 사하자원 有) !== synced(Firestore 사하자원 無)` → 사하자원을 **"신규 로컬"로 오해해 `batch.set`** → Firestore 부활 → 전 기기 동기화로 되살아남 🔴

단일 기기에선 AsyncStorage 도 항상 최신이라 false diff 없음 → **다기기에서만 재현** = 사장님 "서로 물려서" 와 정확히 일치.

## 처방 — Firestore 단일 진실원

- `useOrderPersistence` 가 부팅 시 **addressBook.entries 를 AsyncStorage 에서 복원하지 않음**. meta(todayDate / todayDeliveredKeys / autoRemember)만 복원, entries 는 `prev` 유지(functional update 라 참조 보존 → write effect noop).
- entries 의 유일한 공급원 = Firestore listener. 오프라인은 Firestore `persistentLocalCache`(IndexedDB, `firebase.web.js`)가 마지막 서버 진실을 캐시 → 기기-로컬 stale 이 아니라 **클라우드 진실**을 서빙하므로 안전.
- **데이터 손실 위험 0**: listener 가 안 와도 entries 는 빈 `{}` 로 시작 + ref 도 `{}` → write effect 가 noop(삭제 push 안 함). 최악이라도 잠깐 비었다 곧 채워짐.
