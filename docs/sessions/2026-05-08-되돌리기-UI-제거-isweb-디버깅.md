# 2026-05-08 · 되돌리기 UI 제거 + isWeb ReferenceError 디버깅

> 어제 시작했던 "주문현황·관리자 수익현황의 되돌리기 UI 제거" 작업이 폰 fallback 으로 두 차례 사고 → 오늘 진단 도구 추가 후 진짜 원인(`OrderScreen` 의 잠복 ReferenceError) 잡고 마무리.

## 핵심 Q&A

**Q1. 어제 OTA 발행할 때마다 폰이 fallback 화면 떴다. 변경 코드(되돌리기 UI 제거) 가 원인인가?**
**A1.** 아니다. 진짜 원인은 이틀 전 commit `26a620a` 가 `screens/OrderScreen.js` line 817 에 도입한 `!isWeb` 의 **JSX 평가 시점 ReferenceError**. `isWeb` 정의는 IIFE 콜백 내부에만 있어 외부 스코프에서 미정의. 폰의 native 빌드는 26a620a 이전 코드라 line 817 자체가 없어 정상이었지만, 새 OTA 들어가는 순간 OrderScreen 마운트 즉시 깨짐.

**Q2. 그러면 어제 두 차례 롤백 + 한 차례 이분법은 무의미했나?**
**A2.** 결과만 보면 그렇다. 다만 디버깅 메서드론 합리적인 단계 — 어제 시점엔 진짜 원인 단서 0건. 이분법으로 "KitchenScreen 변경이 원인" 라는 잘못된 결론까진 갔지만, RevenueScreen 무관은 확인했음. 진짜 돌파구는 진단 도구 추가.

**Q3. Sentry 가 왜 stack trace 를 안 잡았나?**
**A3.** `.env` 의 `EXPO_PUBLIC_SENTRY_DSN` 가 빈 값. `utils/sentry.js` 의 `initSentry()` 가 DSN 없으면 silent skip. 폰 fallback 떠도 Sentry capture 안 됨. fallback 화면의 "오류가 자동으로 보고되었습니다" 메시지는 표시만 되고 실제 보고 X.

**Q4. 어떻게 진짜 원인을 찾았나?**
**A4.** `App.js` 의 `CrashFallback` 컴포넌트의 `__DEV__` 가드를 풀어 production 폰에서도 `error.message + error.stack` 첫 6 줄을 화면에 직접 표시. 이걸 OTA 로 한 번 발행 → 다음 폰 fallback 시점에 사장님이 캡처 보냄 → "ReferenceError: Property 'isWeb' doesn't exist at OrderScreen" 한 방에 식별.

**Q5. 수정 방법은?**
**A5.** `OrderScreen` 함수 상단에 `const isWeb = Platform.OS === 'web';` 한 번 정의 + IIFE 콜백 안의 중복 정의 제거. 모든 사용처가 같은 변수 참조. commit `ec27b99`.

**Q6. 어제 의도한 되돌리기 UI 제거는 결국 적용됐나?**
**A6.** 적용. `0605740` 에서 KitchenScreen 의 "최근 조리완료" 섹션 + RevenueScreen 의 "↶ 되돌리기" 버튼 + 미사용 styles 정리. 모든 되돌리기 진입점이 상단 "되돌리기" 탭(UndoScreen) 한 곳으로 단일화. 도메인 로직 / reducer / action 미수정.

**Q7. 운영 영향은?**
**A7.** 어제 사장님 폰이 두 차례 fallback 봤지만 매번 빠른 롤백으로 복구 (각 5분 내). PC 카운터(웹) 도 어제 한 번 0cd27a4 빌드 배포됐지만 PC 영향 보고는 받지 못함 — 사장님이 PC 사용을 평소대로 했어도 OrderScreen 진입 안 했을 가능성, 또는 web 빌드의 minify 가 dead code 처리. (확정 검증 안 함.) 오늘 이후 모든 환경 정상.

**Q8. 같은 사고 재발 방지책은?**
**A8.** 학습 노트 [2026-05-08-isweb-referenceerror-디버깅.md](../learning/2026-05-08-isweb-referenceerror-디버깅.md) 의 "다음 액션" 참고:
1. `EXPO_PUBLIC_SENTRY_DSN` 채워 Sentry 정상화
2. ESLint `no-undef` 룰로 JSX 안 ReferenceError 사전 차단
3. 큰 변경 OTA 전 EAS preview build 로 폰 1회 검증
