## 학습 포인트

### 1. JS 측(라이브 URL) vs .exe 측(NSIS) 빌드 흐름의 분리

매장 PC = .exe wrapper + 라이브 URL JS 코드. 변경 사항 분류:
- **electron/** 디렉토리 (main.js, updater.js, printer/, payment/) → .exe 측. NSIS Setup 받아 설치해야 적용.
- **App.js, screens/, components/, utils/** → JS 측. wrangler deploy 만 하면 라이브 URL 갱신 → 매장 PC 새로고침 시 즉시 적용.

오늘 진단 — wrangler deploy 누락이 1.0.20~1.0.22 의 JS 측 변경을 매장 PC 에 안 닿게 함. 매번 빌드 절차에 wrangler deploy 포함해야.

### 2. electron-updater 의 quitAndInstall 의 한계

`autoUpdater.quitAndInstall(false, true)` 는 .exe 종료가 깨끗해야 작동. 단:
- KIS bridge / SIP listener / 프린터 spool 등 child process 가 살아있으면 .exe 종료 못 함 → silent 실패
- before-quit 핸들러 충돌도 마찬가지

Fix 11 의 8초 timeout 강제 종료 fallback 가 시도였지만 — Fix 11 자체가 .exe 1.0.20+ 부터 활성 → 매장 PC 가 1.0.20 미만이면 해당 코드 자체가 없어서 영원히 silent 실패. 닭-달걀 문제.

→ 결국 자동업데이트 메커니즘 자체를 포기. GitHub 직접 다운로드가 단순/예측 가능.

### 3. autoInstallOnAppQuit:false 의 함정

1.0.11 부터 NSIS hang 방지로 false. 단 그 이후 사장님이 .exe 닫아도 자동 적용 X. "🚀 지금 적용" 버튼이 명시 적용 경로였는데 그게 silent 실패하면 어떤 경로로도 적용 X. 완전 막힘.

→ portable 자동 교체 메커니즘 또는 OS-level scheduler 등 다른 패턴 검토 필요.

### 4. 보수적 패치 + 단계 분할의 가치

1.0.26 에서 메뉴 인라인 편집 Phase A~D 한 번에 (4시간 작업) → categoryBar inline style + PanResponder deps + 빈 화면 사고. 1.0.27 에서 보수적 재작성으로 복구.

향후 큰 변경은 단계 분할 + 매번 검증. 작업 끝나도 매장 PC 에서 빠른 smoke test (라이브 URL 접속 → 빈 화면 안 뜨는지) 후 publish.

### 5. version sync 일관성 (package.json + app.json)

매번 package.json 만 bump 하다가 1.0.27 에서 발견 — 화면 하단 표시 "MyPos v1.0.2 (12)" 가 옛날값. app.json version 이 진짜 사용자 표시값. 두 파일 모두 bump 해야.

```bash
# 매번 같이 bump
sed -i "s/\"version\": \"1.0.X\"/\"version\": \"1.0.Y\"/" package.json
sed -i "s/\"version\": \"1.0.X\"/\"version\": \"1.0.Y\"/" app.json
```
