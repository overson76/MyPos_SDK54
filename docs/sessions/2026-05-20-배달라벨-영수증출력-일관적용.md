## 핵심 기술 결정

### 1. 헬퍼 `resolveDeliveryIdentity` 신설
- **왜**: 3곳(clearTable, clearTableBySource, useAutoClearDelivery) 가 똑같은 패턴으로 alias/phone 추출. 인라인 시 정책 변경 시 3곳 모두 수정 위험.
- **fallback 우선순위**: order 객체에 명시 저장된 값 > 주소록 entry. orderReducer 가 이미 명시 저장 패턴(deliveryAlias/Phone)을 사용 중이라 일관.

### 2. `buildHistoryEntry` 스키마 확장
- **왜**: 재출력 시 영수증 빌더에 alias/phone 전달하려면 history 영속화 단계에서 이미 들어가 있어야 함.
- 옛 history 호환: 새 필드 모두 옵셔널 (`''` / `null` default). 빌더가 빈값이면 자연 주소만 출력.

### 3. RevenueScreen 화면 행 — 단일 라벨 + 보조 주소
- **왜**: 단일 노출 라벨(별칭) 만 보이면 사장님이 회계 확인 시 "어디였더라" 검색 어려움. 별칭/전번 노출 + 주소 한 줄 작은 회색으로 둘 다 보이게.
- 별칭=주소면 보조 줄 안 띄움 (중복 회피).

### 4. `buildOrderSlipText` 는 dead code 지만 함께 패치
- **왜**: 미래에 누가 다시 호출처로 부활시킬 때 정책 일관성 자동 보장.
