# 세션 Q&A — 2026-05-07 (결제완료 / 조리완료 되돌리기 안전망)

주제: 매장 사장님이 결제완료 / 조리완료 버튼을 실수로 눌렀을 때 되돌릴 수 있는 안전망 추가. 매출 history 는 보존하면서 합계만 정정하는 reverted 플래그 패턴.

핵심 결론: **history append-only 원칙 유지 + reverted 플래그로 합계/CSV에서 제외 + 테이블 부활 시 occupied 가드**. UI 두 곳: RevenueScreen 의 이력 row "↶ 되돌리기" 버튼 + KitchenScreen 의 "최근 조리완료" 섹션 (30분 / 10건 캡).

---

## Q&A (질문 순서)

**Q1. 되돌리기가 필요해. 결제완료/조리완료를 실수했을 때 히스토리는 남아 있어야 하고, 되돌리면 해당 테이블이 다시 살아나는 메뉴**  
→ 두 액션의 흐름이 다름:
- **결제완료 (`markPaid`)** = 매출 history 에 entry 가 남음. 되돌리려면 entry 삭제 X — `reverted: true` 플래그 박아 합계에서만 제외 + 테이블 부활.
- **조리완료 (`markReady`)** = 테이블이 살아있는 채로 status 만 'ready' 로 변경. 되돌리려면 단순 토글 (status='preparing', cookState='cooking') — 흔적 안 남김.

설계 제안 후 사장님 동의 → 구현 진행.

**Q2. 그렇게 하면 조리완료하고 사라진 테이블 오더가 다시 주문현황에 나타나?**  
→ 네, 자동. KitchenScreen 의 `status !== 'ready'` 필터([screens/KitchenScreen.js:188](screens/KitchenScreen.js)) 에 다시 통과되면서 같은 selector 가 재계산해서 즉시 메인 그리드에 복귀. 별도 dispatch 불필요. 테이블 데이터 (items, confirmedItems) 는 markReady 시점에 안 지워지고 그대로 살아있어서 추가 복원 작업도 불필요.

**Q3. 응 (구현 진행)**  
→ 5단계 구현:
1. `utils/orderReducer.js` — `orders/undoMarkReady`, `orders/restoreFromHistory` 액션 추가
2. `utils/orderHelpers.js` — `markHistoryReverted(prev, entryId)`, `findHistoryEntry` 헬퍼
3. `utils/OrderContext.js` — `undoMarkReady(tableId)` / `revertHistoryEntry(entryId)` wrapper. 후자는 `{ ok, reason }` 반환 (`notFound` / `occupied` / `alreadyReverted`)
4. `utils/payment.js` — `isCounted` 가드 + 집계 4종 (byPayment / daily / monthly / CSV) 모두 reverted 제외. CSV 는 행 보존 + 합계 0 + "되돌림" 컬럼
5. UI: RevenueScreen 이력 row 빨간 "↶ 되돌리기" 버튼 + reverted 면 회색 배경 + 취소선 + "↶ 되돌림" 라벨. 합계 카드 / 월별 / 결제수단별 / CSV 모두 reverted 제외. KitchenScreen 그리드 하단 "✓ 최근 조리완료" 섹션 — 30분 / 10건 캡, 각 카드에 "↶ 취소" 버튼.

테스트 18케이스 — orderReducer (6) + orderHelpers (7) + payment (5).

**Q4. 지금 작업한 거 아직 깃 안 된 거지?**  
→ 맞음. `git status` 확인 → 10개 파일 modified 상태로 working tree 에만 있음. 커밋 진행할지 물음.

**Q5. 기존에 깃 되어 있는 거 풀 안 한 거 아냐?**  
→ 좋은 지적. 사장님 맥북 ↔ 윈도우 두 PC 환경이라 다른 PC 에서 push 한 새 커밋 가능성 확인. `git fetch origin` → `git rev-list --left-right --count HEAD...origin/main` → `0	0`. 원격에 새 커밋 없음. main / origin/main / 현재 브랜치 모두 동기화. 안전하게 커밋 진행 가능.

**Q6. 진행하고 일단 이 세션은 종료**  
→ 10개 파일 add + commit (`94430cd` — feat(undo): 결제완료 / 조리완료 되돌리기 — 실수 보정 안전망, +541 −17). 브랜치 `claude/blissful-shamir-26e89a` 가 origin/main 보다 1 커밋 앞섬. push 는 사장님 다음 세션 판단으로 진행 권유.

**Q7. 윈도우에서는 권한건너뛰기가 활성화 되어있는데 여기는 안 돼? 깃이라 그런가?**  
→ git / worktree 무관. 원인은 **세션 시작 시점의 권한 모드가 굳어짐**. 윈도우는 시작 시점에 이미 bypassPermissions, 맥북 이번 세션은 default 였음. 해결: `Shift+Tab` 으로 즉시 토글 (이번 세션 한정) + `~/.claude/settings.json` 에 박은 `defaultMode: "bypassPermissions"` 는 다음 세션부터 자동 적용. settings 는 머신별이라 양쪽 PC 따로 박아야.

**Q8. 지금 작업한 거 다 반영되어서 기기에서 실행하면 적용되는 거지?**  
→ 아니. 커밋만 됨. 단계별 필요한 작업:
- 개발용 폰 (`npm start`): main 머지 후 → npm start
- 운영 폰 (TestFlight/Play): main 머지 + `eas update --branch production`
- 매장 PC 카운터 .exe: main 머지 + `npm run deploy:web` → .exe 가 다음 시작 시 새 웹 번들 자동 로드
- 매장 PC .exe 본체 코드: 변경 안 했으니 .exe 새 빌드 불필요

JS 코드만 변경이라 `eas update` + `deploy:web` 두 개로 충분. **deploy:web 는 누적 교훈상 매장 영업 중 PC 연동 풀림 사고 위험** — 사장님 명시 트리거 (`테스트하세요` / `정식 배포`) 받아야 진행.

**Q9. 세션 종료 = 모든 게 반영되게 정리하고 끝나는 게 규칙. 학습노트 + 세션이름 추천 깃 규칙대로**

## 핵심 학습

자세한 패턴 분석은 [docs/learning/2026-05-07-undo-reverted-flag-pattern.md](../learning/2026-05-07-undo-reverted-flag-pattern.md) 참조.

요약:
1. **append-only history 에서 "되돌리기" 는 delete X, reverted 플래그로** — 회계 / 감사 / 운영 분석 흔적 보존.
2. **모든 집계 함수에 `isCounted` 가드 한 줄** — 한 곳 빠뜨리면 합계 ↔ CSV ↔ 화면 어긋남. 단위 테스트로 4곳 모두 검증.
3. **테이블 부활 시 occupied 가드 이중** — wrapper 의 `{ ok: false, reason }` + reducer 의 `if (state[tableId]) return state`. 새 주문 덮어쓰기 사고 방지.
4. **idempotent 보장** — 빠른 더블 클릭 / 동기화 race 시 중복 차감 안 되게 `if (target.reverted) return prev`.
5. **조리완료 = 단순 토글, history 안 만듦** — KitchenScreen 의 selector 가 자동 재계산해서 자연 부활. 새 dispatch 패턴 추가 불필요.

데스크톱 개발 비유: DB 의 `IsCancelled` 컬럼 (UPDATE 안 하고 새 row 추가 안 하고 플래그만 변경) + 모든 쿼리에 `WHERE NOT IsCancelled` 일관되게.
