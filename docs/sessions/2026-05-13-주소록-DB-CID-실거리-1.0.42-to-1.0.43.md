## 알려진 문제 / 미해결 이슈

1. **`AddressBookPanel.confirmEdit` 의 setAddressBook 콜백 안에서 oldKey vs 새 key 매칭 로직이 약간 휴리스틱**: editLabel 호출 직후 setAddressBook 으로 alias/phone 갱신할 때, `editLabel` 이 normalizeAddressKey 로 새 key 를 만들어 entry 를 이주시킨 후라서 oldKey 로 못 찾고, label 매칭으로 폴백. 한 번 더 검토 필요.
2. **75개 임포트 후 75회 카카오 모빌리티 API 호출**: 일 5,000 쿼터 충분하지만, 동시 폭주 가능. 현재 inFlight 가드만 있고 throttle 없음. 한 매장은 큰 문제 X 지만 향후 다매장 운영 시 throttle 검토.
3. **deploy:web → PC .exe 매장 연동 풀리는 이슈**: 1.0.43 기준 자가 복구 카드(c523683+) 로 즉시 복구 가능. 근본 원인(SW + IndexedDB 손상) 아직 미해결. 메모리 `feedback_deploy_breaks_pc_sync.md` 참고.

## 핵심 기술 결정

- **AddressBookModal 과 AddressBookPanel 분리**: 모달은 주문화면 빠른 선택용(기능 가벼움), 패널은 관리 전용(거리/임포트/주소편집/CID 강조 등). 향후 분기될 가능성 높아 한 컴포넌트 재사용보다 분리가 안전.
- **CID 미매칭 → 자동 등록**: 사장님 명시 요구. 잘못 걸려온 전화도 다 등록되는 트레이드오프 받아들임. 직원이 주소록 관리 시 정리.
- **거리는 직선 X, 도로 O**: 사장님 명시 요구. 직선거리는 비현실적이라 운영 판단에 도움 안 됨.
- **API 호출 결과 entry 영구 캐시**: Firestore 자동 동기화. 한 PC 가 호출하면 다른 기기는 무료. 매장 좌표 변경 시만 재계산.
