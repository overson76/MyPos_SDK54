## 핵심 기술 결정
- **무시/식별 키 = 불안정한 내부 key 가 아니라 사용자가 본 안정적 값(상호명).** 대표 key 가 count 로 흔들려도 유지. (진실보석 핵심 교훈)
- **빈 값 거부 가드가 "지우기" 를 막을 때, 삭제 의도는 별도 경로로.** 여기선 주소 삭제 = phone-only 전환(전화로 식별 유지, 기존 CID entry 와 일관).

## 후속 — 무시가 안 저장되던 영속화 레이스 (`c9cc2d9`)
alias 키 처방(`989ad68`) 후에도 사장님 신고 "다른 가게 처리해도 **모달 닫았다 열면 또 뜸**". 진단 질문으로 "누르면 사라지는데 다시 열면 재등장" 확인 → alias 키 자체는 정상(누르면 사라짐), **영속화 실패**가 진짜 원인. `useOrderFirestoreSync` 의 pull(onSnapshot)이 옛 meta(빈 `ignoredSimilarPairs`)로 로컬을 **통째 덮어씀** — 매장 운영 중 다른 기기·다른 필드 변경으로 onSnapshot 이 push 보다 먼저 도착. 처방: pull 시 `ignoredSimilarPairs` 를 **합집합(union) 머지**(`unionPreserveRef` — 로컬이 superset 이면 기존 배열 참조 반환해 write effect 무한 루프 방지). 로컬 무시 보존 + 곧 push 가 Firestore 저장. deploy:web `0a0f25ad` + OTA `c9bafa5e`.

**교훈**: "추가만 하는" 공유 집합(무시 목록 등)은 onSnapshot 이 덮어쓰면 안 됨 → union 머지. 로컬 state 가 즉시 갱신되는데 "닫았다 열면 리셋" = 영속화/sync 레이스 신호.
