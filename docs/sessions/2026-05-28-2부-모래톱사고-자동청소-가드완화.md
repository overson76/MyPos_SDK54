## 핵심 기술 결정

### 1. `isPhoneOnlyLike` 정책 — label === alias

```js
function isPhoneOnlyLike(key, entry) {
  if (key.startsWith('__phone:')) return true;
  const label = (entry?.label || '').trim();
  if (label.startsWith('(주소 미입력)')) return true;
  const alias = (entry?.alias || '').trim();
  if (alias && label === alias) return true;  // 신규
  return false;
}
```

**왜**: 사장님이 `+ 추가` 폼의 주소 칸에 별칭만 입력 → label=alias 박힌 pseudo entry. 진짜 주소 정보 없음. 같은 alias 의 진짜 주소 entry 와 통합 대상.

**안전장치**: `mergeSameAliasPhoneOnlyEntries` 가 *같은 alias 의 정식 entry (label !== alias) 가 있을 때만* 통합. 한 곳뿐인 가게 ("모래톱" 단독 등록) 는 그대로 유지.

### 2. `addAddress` 신규 차단 가드

label=alias 호출 + 같은 alias 의 진짜 주소 entry 발견 → 신규 entry 생성 X, 그 entry 에 phone/request 만 흡수. `console.warn` 로 발생 추적.

**왜**: 청소만 하면 사장님 사용 패턴이 같은 사고 반복 → 신규 발생 자체 차단.

### 3. 부팅 청소 effect 가드 완화

```js
// 변경 전: if (!storeCoord) return;
// 변경 후: storeCoord 없어도 진행. coordInRange 는 storeCoord 없으면 true 가정(좌표 reset X),
//          straightKm 은 null, isDrivingMSane(drivingM, null) 이 절대 임계(10km) 만으로 reject.
```

**왜**: 매장 좌표 미설정 매장도 잘못된 drivingM (10km 초과 잔재) 자동 청소 가능. 좌표 정상성 판단은 storeCoord 필요해서 그대로.
